6-1 I/O 接口

基础
🔑 核心问题：CPU 如何"看到"并"访问"I/O 设备？
访问 I/O 设备的两种方式
  • Memory-Mapped I/O（内存映射） — I/O 与内存共享同一地址空间，用 Load/Store 访问
  • ✅ 任何访存指令都能操作 I/O   ❌ 占用内存地址空间
  • Isolated I/O（独立编址） — I/O 与内存地址空间分离，专用指令（IN/OUT）
  • ✅ I/O 设备仅需少量地址线   ❌ 需要额外指令
I/O 设备接口
  • 位于设备与互连网络之间的电路，提供数据传输与状态/控制信息交换
  • 包含三类寄存器：数据寄存器（Data）、状态寄存器（Status）、控制寄存器（Control）

6-2 程序控制 I/O（Polling）

轮询方式
🔑 核心问题：最简单的 I/O 方式如何工作？代价是什么？
基本原理
  • CPU 在循环中反复查询设备状态寄存器中的 Ready 标志位
  • 设备通过状态标志（如键盘 KIN、显示器 DOUT）表示"数据已就绪"
  • CPU 检测到 Ready=1 后才执行数据读写
等待循环实现
  • 键盘读取：轮询 KBD_STATUS，检查 KIN 标志，然后读 KBD_DATA
  • 显示器输出：轮询 DISP_STATUS，检查 DOUT 标志，然后写 DISP_DATA
优缺点
  • ✅ 硬件最简单，易于理解和实现
  • ❌ CPU 大量时间浪费在"忙等"（busy waiting），无法执行有用计算

6-3 中断概念与处理

中断机制
🔑 核心问题：如何让 CPU 不再空等，由设备主动通知？
中断定义
  • 中断：导致 CPU 暂停当前程序执行、转而执行中断服务程序（ISR）的事件
  • 典型场景：定时器每 10 秒中断一次，CPU 暂停 COMPUTE 例程去执行 DISPLAY 例程
核心概念
  • 中断请求（Interrupt Request） — I/O 设备通过总线控制线向 CPU 发送的信号
  • 中断应答（Interrupt Acknowledge） — CPU 发出的确认信号
  • 中断服务程序（ISR） — 响应中断而执行的例程
  • 中断延迟（Interrupt Latency） — 从收到中断请求到开始执行 ISR 之间的时间
  • ISR vs 普通子程序 — ISR 与被中断程序无关联，必须先保存现场（PC、PSW、寄存器），完成后恢复
中断处理流程
  • 接收中断请求 → 保存现场（PC、PSW）→ 执行 ISR → 恢复现场 → 返回被中断程序
  • 大多数现代处理器只自动保存最小上下文（PC + PSW），其余由软件保存
中断的使能与禁止（两级控制）
  • 处理器端：PSR 中的 IE（Interrupt Enable）位，IE=0 时忽略所有中断
  • 设备端：设备接口控制寄存器中的中断使能位，控制单个设备能否发中断

6-4 多设备中断系统设计

系统设计
🔑 核心问题：多个设备都要中断 CPU 时，如何识别、嵌套、仲裁？
问题一：识别中断源
  • 多中断请求线 — 每个设备/每组设备独立的中断线
  • 非向量中断（软件轮询） — CPU 跳到固定地址，逐一查询各设备状态寄存器
  • 向量中断（Vectored Interrupt） — 设备主动发送中断向量码，通过中断向量表找到对应 ISR 地址
问题二：中断嵌套（Multi-level Interrupt）
  • 为设备和处理器分配优先级，高优先级设备可中断低优先级 ISR
  • 处理器优先级编码在 PSW 中，只接受比自身优先级高的中断
  • 进入 ISR 时将处理器优先级提升至该设备优先级
问题三：同时中断的仲裁
  • 独立中断线 — 硬件优先级仲裁电路，选择最高优先级
  • 软件轮询 — 按轮询顺序决定优先级
  • 菊花链（Daisy Chain） — 离 CPU 电气距离越近的设备优先级越高
  • 优先级分组 — 将设备分配到不同优先级级别，每组内再用菊花链或轮询

6-5 直接存储器访问（DMA）

高速传输
🔑 核心问题：高速大块数据如何彻底绕开 CPU 直接传输？
DMA 的动机
  • 程序控制 I/O 和中断驱动 I/O 都要求 CPU 介入每一个字节的传输
  • 传输速率受限，CPU 被绑定在 I/O 管理上
DMA 工作原理
  • DMA 控制器（DMAC） 接管系统总线，直接在 I/O 设备与主存之间传输数据块
  • 三步流程：① CPU 准备（告知设备地址、内存地址、块大小、方向、模式）→ ② DMAC 控制数据传送（CPU 继续其他工作）→ ③ DMAC 发出中断通知完成
DMA 控制器功能
  • 提供内存地址、递增地址、计数已传字节、产生所有总线控制信号
  • 内部寄存器：起始地址、字计数、状态/控制寄存器
三种传输模式
  • Burst（突发）模式 — 一次性传输整个块。✅ 适合加载程序/文件   ❌ CPU 长时间挂起
  • Cycle Stealing（周期窃取） — 每次只传一个字节就归还总线，反复请求。✅ CPU 使用率更高
  • Transparent（透明）模式 — 仅在 CPU 不使用总线时传输。✅ CPU 完全无感   ❌ 硬件复杂

📊 三种 I/O 通信方式对比

维度 程序控制 I/O (Polling) 中断驱动 I/O (Interrupt) DMA
CPU 角色 全程轮询等待设备就绪 设备通知，CPU 介入每字节传输 仅准备与收尾，传输过程不参与
CPU 效率 低（空转忙等） 中（可做其他工作） 高（完全解放）
数据传输单位 字/字节 字/字节 块（block）
适用场景 简单嵌入式系统 低速字符设备（键盘、鼠标） 高速块设备（磁盘、网卡、显卡）
硬件复杂度 最低 中等 最高（需 DMA 控制器）
启动方式 CPU 主动轮询 设备主动发中断请求 CPU 初始化 DMAC 后由 DMAC 控制
